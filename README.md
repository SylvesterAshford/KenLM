## 2. How to run on your local machine

You need two things:

1. **The Python `kenlm` binding** — used to *score* text (compute perplexity).
2. **The KenLM CLI binaries `lmplz` and `build_binary`** — used to *train* the LM. The Python package alone cannot train; the binaries must be built from source.

### 2.1 macOS (Apple Silicon / Intel)

Tested on macOS 14+, Homebrew, Anaconda.

#### Step A — install build dependencies via Homebrew
```bash
brew install cmake boost eigen
```

#### Step B — install Python packages
```bash
pip install kenlm matplotlib nbformat jupyter
```

#### Step C — clone and build KenLM
```bash
cd ~/Desktop
git clone https://github.com/kpu/kenlm.git
cd kenlm && mkdir build && cd build
cmake .. -DCMAKE_BUILD_TYPE=Release
make -j4
```

#### Step D — patch one line in KenLM (Boost 1.87+ compatibility)

If `cmake` fails with `Could not find a package configuration file provided by "boost_system"`, edit `~/Desktop/kenlm/CMakeLists.txt` around line 100 and remove the `system` component — Boost 1.87+ made `boost_system` header-only, so it's no longer a separate library:

```cmake
find_package(Boost 1.41.0 REQUIRED COMPONENTS
  program_options
  thread
  unit_test_framework
)
```

Then clear the build cache and reconfigure:
```bash
cd ~/Desktop/kenlm/build
rm -rf *
cmake .. -DCMAKE_BUILD_TYPE=Release
make -j4
```

#### Step E — launch the notebook with `KENLM_BIN` set
```bash
cd ~/Desktop/class-15
KENLM_BIN=~/Desktop/kenlm/build/bin jupyter notebook class-15.ipynb
```

Or make the env var permanent in `~/.zshrc`:
```bash
echo 'export KENLM_BIN=~/Desktop/kenlm/build/bin' >> ~/.zshrc
source ~/.zshrc
```

#### Step F — verify
The first executable cell of the notebook should print:
```
KenLM binaries: /Users/<you>/Desktop/kenlm/build/bin
kenlm Python binding: /opt/anaconda3/.../kenlm/__init__.py
```

## 3. Concepts you need to understand the notebook

### 3.1 Language Model (LM)

A language model assigns probabilities to sequences of tokens. For a sentence `w₁ w₂ … wₙ` an n-gram LM factors that probability as:

```
P(w₁ … wₙ) ≈ Π P(wᵢ | wᵢ₋ₙ₊₁ … wᵢ₋₁)
```

A **3-gram (trigram) LM** estimates each token given the previous two. KenLM is a fast, memory-efficient n-gram toolkit by Kenneth Heafield.

### 3.2 Perplexity (PPL)

Perplexity = the model's average per-token uncertainty, exponentiated:

```
PPL(text) = 2^( -1/N · Σ log₂ P(wᵢ | context) )
```

- **Lower PPL = better fit.** A PPL of 50 roughly means the model is "about as confused as if it had 50 equally-likely choices per token."
- PPL is sensitive to vocabulary and tokenization, so you can only compare PPLs of models trained on the *same* token scheme.

### 3.3 Modified Kneser–Ney smoothing

Real n-gram counts are sparse — most trigrams never appear. **Smoothing** redistributes probability mass from observed to unobserved n-grams. *Modified Kneser–Ney* (Chen & Goodman, 1998) is the de-facto standard for n-gram LMs and is what `lmplz` uses.

`--discount_fallback` is required only when the corpus is tiny (some discount estimates underflow); drop the flag for real corpora.

### 3.4 Syllable tokenization with **sylbreak**

Myanmar script has no spaces between words, but it does have a regular syllable structure. Ye Kyaw Thu's [sylbreak](https://github.com/ye-kyaw-thu/sylbreak) regex segments text at every Myanmar consonant **unless** that consonant is preceded by virama (`္`) or followed by virama / asat (`်`). Syllables are a stable, language-appropriate token unit for Myanmar LMs.

### 3.5 Domain mismatch and adaptation

A model trained on general prose will score poorly on text whose vocabulary, syntax, or register differs — **out-of-domain text → high PPL**. For Myanmar:

| Domain | Why it's hard for a general LM |
|---|---|
| Religious (gāthā) | Pali loans (သီလ, မေတ္တာ, နိဗ္ဗာန) that don't appear in general prose |
| Legal | Formal vocabulary (ပုဒ်မ, တရားရုံး, ပြစ်ဒဏ်), highly nominal syntax |
| Facebook | Colloquial particles (ဟေ့, ဪ, နော်), slang, abbreviations |

### 3.6 Linear interpolation (the adaptation method used here)

Train a *small* in-domain LM `P_dom`. Combine its probabilities with the base LM `P_base`:

```
P_mix(w | h) = λ · P_dom(w | h) + (1 − λ) · P_base(w | h)
```

Sweep λ ∈ {0.1, …, 0.7} to find the value that minimises PPL. Conceptually: when λ=0 we trust the base LM only; when λ=1 we trust the tiny domain LM only. The sweet spot blends both.

## 4. Code walkthrough — section by section

The notebook is divided into eight sections. Each one maps roughly to one cell or a small group of cells.

### Section 0 — Setup

Imports, defines `ROOT/DATA/MODELS/OUT` paths, reads `KENLM_BIN` from the environment, and asserts that `lmplz` exists. If this cell fails you have either not set `KENLM_BIN` or not built the binaries — see Section 2 above.

### Section 1 — Corpus inspection

Reads `data/train/general_corpus.txt` and prints character count, line count, and a sample. The shipped corpus has ~200 multi-topic sentences (geography, family, education, technology, etc.) — deliberately not skewed toward any single domain.

### Section 2 — Cleaning + tokenization (sylbreak)

Two helpers:

- `clean(text)` — strips Myanmar punctuation (`။`, `၊`), Latin/digit runs, and ASCII/Unicode punctuation. Collapses whitespace.
- `sylbreak(text)` — applies the regex described in §3.4 and returns whitespace-separated syllables.

The `tokenize_file(in_path, out_path)` function streams a file line-by-line, runs `clean` then `sylbreak`, and writes the syllable-segmented output. Run on `general_corpus.txt` it produces `general_corpus.syl.txt`.

### Section 3 — Train the base LM

Runs `lmplz -o 3 --discount_fallback -S 30%` as a subprocess, piping the syllable-tokenized training data on stdin and writing the ARPA file on stdout. Then runs `build_binary` to compile the ARPA into a memory-mapped `.bin` file for fast loading. Finally, loads the binary with `kenlm.Model(...)`.

- `-o 3` — 3-gram order.
- `-S 30%` — use up to 30% of system memory during training.
- `--discount_fallback` — needed because the corpus is tiny.

### Section 4 — Domain test sets

Tokenizes the three `*_raw.txt` files, then trims each to **exactly 10 sentences × 20 syllables = 200 syllable tokens**. This identical length is essential for fair PPL comparison across domains.

### Section 5 — PPL of the base LM per domain

For each test file, calls `base_lm.perplexity(text)` (KenLM adds `<s>` / `</s>` and averages logprob over tokens). Also reports an in-domain PPL on a slice of the training data as a sanity floor. Plots a log-scale bar chart and saves it to `outputs/base_ppl_by_domain.png`.

### Section 6 — Adaptation brainstorm

A markdown table comparing five adaptation strategies (mix-in, linear interpolation, vocabulary expansion, class-based, neural fine-tuning). The notebook implements **linear interpolation** because it is fast, classical, and well-understood.

### Section 7 — Adaptation experiment

For each domain:

1. **`train_domain_lm`** holds out the first 200 syllables (the test portion) and trains a tiny 2-gram LM on the rest of the same-domain file.
2. **`interp_ppl`** iterates token-by-token through the test text, retrieves `log P` from both models via `full_scores()`, converts to linear probability, mixes with weight λ, and computes PPL.
3. Sweeps λ over `{0.1, …, 0.7}` and records the best λ per domain.
4. Plots the λ-sweep curve (`outputs/lambda_sweep.png`) and a before/after bar chart (`outputs/before_after.png`).

The adaptation corpus per domain is tiny — only the held-out tail of the test file. With real in-domain text the improvement would be larger.

## 5. Results

Expected output on the shipped sample data:

| Domain | Base PPL | Adapted PPL | Reduction |
|---|---:|---:|---:|
| Religious (gāthā) | 845.1 | 92.6 | −89% |
| Legal | 216.8 | 57.6 | −73% |
| Facebook | 530.1 | 90.9 | −83% |

Hardest domain for the base LM: **religious text** (Pali-loan vocabulary), as expected. Linear interpolation with even a tiny in-domain LM closes most of the gap.

## 7. Troubleshooting

**`AssertionError: lmplz not found at ../Desktop/kenlm/build/bin/`**
Jupyter wasn't launched with `KENLM_BIN` set. Either restart with `KENLM_BIN=~/Desktop/kenlm/build/bin jupyter notebook class-15.ipynb`, or hard-code the absolute path in the setup cell.

**`Could not find a package configuration file provided by "boost_system"`**
You're on Boost 1.87+. Patch `kenlm/CMakeLists.txt` to remove the `system` line as shown in §2.1 Step D.

**`Could not find Eigen3 ... version not compatible`**
Harmless warning — Eigen is optional for KenLM. The build proceeds.

**Notebook fails on `lmplz` exit code != 0**
Most often a tokenization or empty-input problem. Inspect the printed stderr tail and confirm `general_corpus.syl.txt` is non-empty.

**`pip install kenlm` fails to compile**
Install boost/eigen via Homebrew first, then retry pip — the wheel needs C++ headers.
